Here’s a practical way to debug a complex React app that uses **TanStack Query**, **dnd-kit**, and **nuqs**, especially when there are lots of dependent components.


---


# 1. First: narrow the problem


Before adding tools, define:


- **What is broken?**
  - wrong UI
  - stale data
  - drag/drop not updating
  - query not refetching
  - URL state not syncing
  - excessive rerenders
  - race condition
- **When does it happen?**
  - initial load
  - after drag
  - after mutation
  - after query invalidation
  - after URL param change
- **Can you reproduce it consistently?**
  - exact steps
  - exact route
  - exact params
  - expected vs actual result


Write this down. Even short notes help:
- “Dragging item A to column B updates UI, but after refetch it jumps back.”
- “Changing nuqs search param updates URL, but query does not refetch.”


That gives you a target.


---


# 2. Build a debugging strategy: isolate layers


With your stack, the bug is usually in one of these layers:


1. **URL state layer** — `nuqs`
2. **server/cache layer** — TanStack Query
3. **view state layer** — local React state / props / derived state
4. **interaction layer** — dnd-kit events
5. **rendering layer** — memoization, keys, stale closures, rerenders


Debug layer by layer instead of the whole system at once.


A useful question is:


> “At what exact point does the data become wrong?”


For example:
- URL param is correct
- query key is wrong
- query response is correct
- transformed data is wrong
- drag event payload is wrong
- rendered list keys are wrong


Once you know where it first becomes wrong, you’ve found the root cause area.

# 4. Start with a minimal reproducible path


Don’t debug the entire app at once.


Create the smallest scenario that still reproduces the issue:
- one page
- one query
- one drag interaction
- one URL param
- one mutation


If needed:
- hardcode response data
- disable unrelated queries
- render only the list in question
- remove styling/animations temporarily


This helps answer:
- Is the bug in data flow?
- Is the bug in drag/drop?
- Is the bug in synchronization between query and URL?
- Is the bug caused by another component side effect?

# 5. Add structured logging, not random logging


Don’t just do:
```ts
console.log(data)
```


Use logs that answer:
- where
- when
- what changed
- what triggered it


Example utility:


```ts
export function debugLog(scope: string, message: string, payload?: unknown) {
  const time = new Date().toISOString();
  console.log(`[${time}] [${scope}] ${message}`, payload ?? "");
}
```


Use it like:
```ts
debugLog("BoardPage", "render", { searchParams, selectedColumnId });
debugLog("useItemsQuery", "query key", queryKey);
debugLog("dnd", "drag end", { activeId, overId });
debugLog("mutation", "onSuccess", result);
```


Better yet, use grouped logs:


```ts
export function debugGroup(label: string, fn: () => void) {
  console.group(label);
  try {
    fn();
  } finally {
    console.groupEnd();
  }
}
```


Example:
```ts
debugGroup("[DragEnd]", () => {
  console.log("active", active);
  console.log("over", over);
  console.log("current items", items);
});
```

# 6. Track renders explicitly


In complex component trees, you need to know:
- which component rerendered
- why it rerendered
- what prop changed


Utility hook:


```ts
import { useEffect, useRef } from "react";


export function useRenderLogger(name: string, props: Record<string, unknown>) {
  const prev = useRef(props);


  useEffect(() => {
    const changedProps = Object.entries(props).reduce<Record<string, { before: unknown; after: unknown }>>(
      (acc, [key, value]) => {
        if (prev.current[key] !== value) {
          acc[key] = { before: prev.current[key], after: value };
        }
        return acc;
      },
      {}
    );


    if (Object.keys(changedProps).length > 0) {
      console.log(`[render] ${name}`, changedProps);
    }


    prev.current = props;
  });
}
```


Usage:
```ts
useRenderLogger("TaskList", { items, columnId, isDragging });
```


This is very helpful for:
- unstable arrays/objects
- callback identity changes
- rerender storms

# 7. Debug TanStack Query specifically


For TanStack Query, root causes are often:


## A. Wrong query keys
Make sure query keys include all dependencies.


Bad:
```ts
useQuery({
  queryKey: ["items"],
  queryFn: () => fetchItems(filter),
});
```


Better:
```ts
useQuery({
  queryKey: ["items", filter],
  queryFn: () => fetchItems(filter),
});
```


If `filter` comes from `nuqs`, ensure the query key reflects it.


---


## B. Query key values are unstable
If you pass a new object every render, keys can behave unexpectedly.


Prefer:
```ts
const queryKey = ["items", status, sort];
```


Instead of:
```ts
const queryKey = ["items", { status, sort }];
```


Unless you are sure object shape and serialization are stable.


---


## C. Mutation updates cache incorrectly
If drag/drop changes order, inspect:
- `onMutate`
- optimistic updates
- `onSuccess`
- `onError`
- `invalidateQueries`


Common bug:
- optimistic UI reorders correctly
- server response differs
- invalidation restores old order
- or stale cache overwrites new state


Log all mutation lifecycle stages.


Example:
```ts
const mutation = useMutation({
  mutationFn: updateOrder,
  onMutate: async (variables) => {
    console.log("[mutation:onMutate]", variables);
  },
  onSuccess: (data) => {
    console.log("[mutation:onSuccess]", data);
  },
  onError: (error) => {
    console.error("[mutation:onError]", error);
  },
  onSettled: () => {
    console.log("[mutation:onSettled]");
  },
});
```


---


## D. Derived local state from query data
This is a major source of bugs.


Example:
```ts
const { data } = useQuery(...);
const [items, setItems] = useState(data ?? []);
```


This can desync because `items` won’t automatically stay aligned with query data.


If you do this, ask:
- why do I need local state?
- should I derive directly from query data?
- should local state reset when query data changes?


If local state is necessary, explicitly sync it:
```ts
useEffect(() => {
  if (data) setItems(data);
}, [data]);
```


But even then, be careful not to overwrite optimistic drag state unexpectedly.


---


# 8. Debug dnd-kit specifically


For dnd-kit, common issues are:


## A. Wrong IDs
Ensure draggable and droppable ids are:
- unique
- stable
- same type consistently (`string` vs `number`)


A lot of bugs come from:
- `"1"` vs `1`
- index-based IDs
- changing IDs after reorder


---


## B. `over` is null
This happens often.


In `onDragEnd`, log:
```ts
const handleDragEnd = (event) => {
  const { active, over } = event;
  console.log("[dragEnd]", {
    activeId: active?.id,
    overId: over?.id,
    activeData: active?.data?.current,
    overData: over?.data?.current,
  });
};
```


Check:
- Is the drop target registered?
- Is layout shifting during drag?
- Is collision detection strategy appropriate?


---


## C. State update logic after drag is wrong
Verify:
- source container
- target container
- source index
- target index


Create a pure helper function and test it separately:


```ts
export function moveItemBetweenLists({
  sourceItems,
  targetItems,
  sourceIndex,
  targetIndex,
}) {
  const sourceClone = [...sourceItems];
  const targetClone = [...targetItems];
  const [moved] = sourceClone.splice(sourceIndex, 1);
  targetClone.splice(targetIndex, 0, moved);


  return {
    sourceItems: sourceClone,
    targetItems: targetClone,
  };
}
```


Then test this helper independently. Pure functions are much easier to debug than UI handlers.


---


## D. Drag overlay / measuring / collision settings
If dragging behaves strangely, inspect:
- `collisionDetection`
- `modifiers`
- `DragOverlay`
- scroll containers
- nested droppables


Temporarily simplify:
- remove overlay
- use simpler collision detection
- remove modifiers
- test one sortable list only


---


# 9. Debug nuqs specifically


For `nuqs`, bugs usually happen when URL state and query state drift apart.


Check:


## A. Is the URL param actually updating?
Log:
```ts
console.log("search params changed", value);
```


Also inspect browser URL directly.


---


## B. Is the parsed value what you expect?
If you use parsers, verify output type:
- string
- number
- array
- nullable value


Example issue:
- URL has `"1"`
- parser gives `1`
- elsewhere you compare with string `"1"`
- query key or dnd id mismatch happens


---


## C. Does the query depend on the param?
If a URL param changes but query doesn’t react, the param may not be included in:
- query key
- query fn dependency
- derived state dependency


---


## D. Is there a default value masking the issue?
Sometimes a parser/default makes it look like state changed when it didn’t.

# 10. Make event timelines visible


In a complex app, logs should show event order.


For example, when a drag ends:


1. drag end event fires
2. local reorder happens
3. mutation starts
4. URL param changes?
5. query invalidates
6. query refetch returns
7. local state resets
8. UI jumps


That timeline often reveals the exact bug.


You can implement a simple timeline logger:


```ts
let debugCounter = 0;


export function trace(scope: string, event: string, payload?: unknown) {
  debugCounter += 1;
  console.log(`#${debugCounter} [${scope}] ${event}`, payload ?? "");
}
```


Example:
```ts
trace("dnd", "dragEnd", { activeId, overId });
trace("board", "local reorder applied", reorderedItems);
trace("mutation", "start", variables);
trace("mutation", "success", response);
trace("query", "invalidate items");
trace("query", "new data", data);
```


This makes ordering obvious.

# 11. Convert complex logic into pure utility functions


A huge debugging improvement is to move logic out of components.


Extract:
- reorder logic
- query key builders
- URL param parsers/adapters
- server payload transformers
- selectors / derived data calculators


Example:
```ts
export function getItemsQueryKey(filters: {
  boardId: string;
  status?: string;
  sort?: string;
}) {
  return ["items", filters.boardId, filters.status ?? "all", filters.sort ?? "default"];
}
```


Benefits:
- easier to log
- easier to unit test
- easier to compare inputs/outputs
- less hidden component behavior

# 12. Add assertions for impossible states


In complex systems, fail early.


Example:
```ts
export function invariant(condition: unknown, message: string): asserts condition {
  if (!condition) {
    throw new Error(message);
  }
}
```


Use:
```ts
invariant(over, "Expected drop target 'over' to exist on drag end");
invariant(typeof active.id === "string", "Expected active.id to be a string");
```


This helps catch broken assumptions immediately instead of causing weird downstream UI bugs.

# 13. Compare “source of truth” carefully


For your stack, ask for every piece of state:


- Is the source of truth the **URL**?
- the **query cache**?
- **local component state**?
- the **server**?
- the **drag interaction temporary state**?


Many bugs come from having multiple sources of truth.


Example anti-pattern:
- filter exists in URL
- also in local state
- also affects query key
- also copied into a child component state


That creates synchronization bugs.


Prefer one clear owner for each state value.

# 14. Check for common React pitfalls


In complex dependent components, these are frequent causes:


## Unstable object/array props
```ts
<MyComp options={{ sort, filter }} />
```


This creates a new object every render.


---


## Stale closures
```ts
const handleDragEnd = useCallback(() => {
  console.log(items); // maybe stale
}, []);
```


Missing dependencies can cause old state to be used.


---


## Wrong keys in lists
Never use array index if order changes.


Bad:
```tsx
{items.map((item, index) => (
  <Row key={index} item={item} />
))}
```


Use stable IDs:
```tsx
{items.map((item) => (
  <Row key={item.id} item={item} />
))}
```


This is especially important with dnd-kit.


---


## State syncing effects causing loops
```ts
useEffect(() => {
  setState(derive(data));
}, [data]);
```


This may overwrite user interactions or create repeated updates.

# 18. A practical root-cause workflow


Here’s a repeatable process:


## Step 1: Reproduce reliably
Record exact steps.


## Step 2: Inspect visible wrong output
What exactly is wrong in UI?


## Step 3: Trace backward
Check:
- rendered props
- local state
- derived state
- query cache
- URL params
- drag event payload
- server response


Find the first place where data becomes wrong.


## Step 4: Verify assumptions
Examples:
- Is `active.id` really what I think?
- Is `queryKey` really changing?
- Is cache really invalidated?
- Is local state overwriting fresh query data?


## Step 5: Isolate
Remove unrelated code until bug disappears.


## Step 6: Confirm fix with instrumentation
After fixing, keep logs/devtools long enough to verify:
- expected event order
- expected query cache updates
- expected rerender behavior

Debugging a complex React application with **TanStack Query (data fetching)**, **dnd-kit (drag and drop)**, and **nuqs (URL search param state)** is challenging because these three libraries control different "sources of truth." 


When a bug occurs, it’s usually because these states fall out of sync or trigger infinite re-rendering loops (e.g., dropping an item updates the URL, which triggers a query fetch, which re-renders the drag context and cancels the drag).


Here is a comprehensive step-by-step guide, along with the tools, extensions, and utility functions you need to find the root cause.


---


### Phase 1: Essential DevTools & Setup


Before you start hunting, equip your browser and codebase with the right tools.


1.  **React Developer Tools (Browser Extension)**
    *   **Why:** Essential for checking if `dnd-kit` components are unmounting unexpectedly or if `nuqs` state changes are causing massive re-renders.
    *   **Action:** Enable "Highlight updates when components render" in the settings. If your whole screen flashes when you drag an item, you have a re-render issue.
2.  **TanStack Query DevTools**
    *   **Why:** You need to see exactly what your queries and mutations are doing.
    *   **Action:** Install `@tanstack/react-query-devtools` and add `<ReactQueryDevtools initialIsOpen={false} />` to your root provider. Watch for queries that are constantly in a "fetching" or "stale" state.
3.  **Disable React StrictMode (Temporarily)**
    *   **Why:** StrictMode double-invokes components in development. When debugging complex `dnd-kit` drag events or `nuqs` URL parsing, this can create "ghost" logs that confuse you. Turn it off in `next.config.js` or `main.tsx` *just until you find the bug*.


---


### Phase 2: Step-by-Step Debugging Strategy


Don't look at the whole app. Break the problem down into the intersection of your three libraries.


#### Step 1: Isolate the source of truth
Ask yourself: *Where does the bug start?*
*   Does it happen on page load? (Likely **nuqs** + **TanStack Query** interaction).
*   Does it happen when moving an item? (Likely **dnd-kit**).
*   Does it happen *after* dropping an item? (Likely **dnd-kit** triggering a **Query Mutation**).


#### Step 2: Test the libraries in a vacuum
*   **Mock the Network:** Disconnect your internet or mock the TanStack Query response. Does the UI still break when you drag? If no, the bug is in your mutation/invalidation logic.
*   **Remove URL syncing:** Temporarily replace your `nuqs` hooks (`useQueryState`) with standard `useState`. Does the drag-and-drop work now? If yes, the bug is caused by the URL update triggering a Next.js/React router navigation that unmounts your components.


#### Step 3: Analyze the "Drop" Flow (The most common failure point)
Usually, the bug lives in this exact sequence:
1. `onDragEnd` fires.
2. You call a TanStack mutation to update the backend.
3. You update the `nuqs` URL state.
4. TanStack Query invalidates and refetches.
**If components remount or lose their `id` props during steps 2-4, dnd-kit will crash.**


---


### Phase 3: Common Stack-Specific Gotchas to Look For


*   **dnd-kit:** 
    *   Ensure the `id` you pass to `useDraggable` and `useDroppable` is **100% stable**. If a TanStack query refetches and changes the IDs, the drag will fail.
    *   Ensure your Sortable items are wrapped in `React.memo()`.
*   **nuqs:** 
    *   If you are updating the URL on every pixel a user drags an item, you will hit the browser's History API rate limit (50 updates per second) or cause massive lag. Ensure you only update `nuqs` state `onDragEnd`.
    *   Use `{ shallow: true }` in `nuqs` to prevent full page reloads when the URL changes.
*   **TanStack Query:** 
    *   Are you using `useSuspenseQuery`? If `nuqs` updates a search param that changes a Query Key, the component might suspend, showing a loading fallback and destroying your `dnd-kit` `DndContext`.


---


### Phase 4: Utility Functions & Logging Methods


Add these snippets to your codebase to track down exactly what is happening.


#### 1. The "Why Did You Render?" Hook
Because you have many dependent components, you need to know *which* prop or state change triggered a render. Drop this into your heaviest components (like your Kanban board or List container).


```javascript
import { useEffect, useRef } from 'react';


export function useTraceUpdate(componentName, props) {
  const prev = useRef(props);
  useEffect(() => {
    const changedProps = Object.entries(props).reduce((ps, [k, v]) => {
      if (prev.current[k] !== v) {
        ps[k] = [prev.current[k], v];
      }
      return ps;
    }, {});
    
    if (Object.keys(changedProps).length > 0) {
      console.log(`[Re-render] ${componentName} changed props:`, changedProps);
    }
    prev.current = props;
  });
}


// Usage inside your component:
// useTraceUpdate('SortableItem', { id, isDragging, urlParam, queryData });
```


#### 2. Grouped Action Logging
When a drag ends, many things happen. Group them in your console so they don't get lost in the noise.


```javascript
const handleDragEnd = (event) => {
  const { active, over } = event;
  
  console.groupCollapsed('🛑 Drag End Flow Sequence');
  console.log('1. dnd-kit event:', { active: active.id, over: over?.id });
  
  if (!over) {
    console.warn('Dropped outside a droppable area');
    console.groupEnd();
    return;
  }


  console.log('2. Optimistic UI Update (Local State)');
  // ... your optimistic update code ...


  console.log('3. Triggering TanStack Mutation');
  mutation.mutate({ id: active.id, newParentId: over.id }, {
    onSuccess: () => {
      console.log('  -> Mutation Success');
    },
    onError: (err) => {
      console.error('  -> Mutation Failed', err);
    }
  });


  console.log('4. Updating nuqs URL State');
  setSearchParams({ activeId: null }); 


  console.groupEnd();
};
```


#### 3. Monitor TanStack Query Cache Subscriptions
Sometimes, a component rerenders because it's subscribed to a query that is fetching in the background. You can log query lifecycles globally:


```javascript
// In your queryClient setup
import { QueryClient } from '@tanstack/react-query';


const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onSuccess: (data, query) => {
      console.log(`[Query Cache] ✅ Updated: ${query.queryKey.join('-')}`, data);
    },
    onInvalidate: (query) => {
      console.log(`[Query Cache] 🔄 Invalidated: ${query.queryKey.join('-')}`);
    }
  }),
});
```


### Summary Checklist to find the bug:
1. [ ] Check React DevTools: Are `DndContext` or `SortableItem` components unmounting during drag?
2. [ ] Check TanStack DevTools: Is a query constantly refetching?
3. [ ] URL Check: When the bug happens, does the URL change? If yes, `nuqs` is likely triggering a hydration mismatch or Next.js navigation.
4. [ ] Implement `useTraceUpdate` on the component that crashes to see exactly which prop (from query, dnd, or nuqs) broke it.
Debugging a complex React application using **TanStack Query**, **dnd kit**, and **nuqs** can be tricky because you are dealing with three very different types of state: **Server State** (TanStack), **Transient UI/Interaction State** (dnd kit), and **URL State** (nuqs). 


When these three interact, the most common root causes of bugs are **unintended re-renders**, **stale closures**, and **loss of DOM refs**.


Here is a comprehensive, step-by-step guide, including tools, utility functions, and stack-specific gotchas to help you find the root cause.


---


### Step 1: Equip the Right Extensions & DevTools


Before adding `console.log` everywhere, let the official tools do the heavy lifting:


1.  **TanStack Query Devtools**: 
    *   *Why:* Essential to see if queries are stuck in `fetching`, `stale`, or `inactive`. 
    *   *Setup:* Install `@tanstack/react-query-devtools` and add `<ReactQueryDevtools initialIsOpen={false} />` to your root.
2.  **React Developer Tools (Browser Extension)**:
    *   *Why:* Use the **Profiler** tab. Check the "Record why each component rendered while profiling" setting in the gear icon. This is vital for seeing if URL changes (nuqs) are causing your drag context (dnd kit) to unmount.
3.  **Why Did You Render (WDYR)**:
    *   *Why:* A library that monkey-patches React to notify you in the console when avoidable re-renders occur.
    *   *Setup:* Install `@welldone-software/why-did-you-render`.

### Step 2: Implement Strategic Logging Utilities


Generic `console.log` gets lost in complex apps. Use these specific utility functions to track state transitions.


#### A. The "Why Did This Re-render" Hook
Use this hook inside components that frequently unmount or break during drag-and-drop.
```javascript
import { useEffect, useRef } from 'react';


export function useTraceUpdate(componentName, props) {
  const prev = useRef(props);
  useEffect(() => {
    const changedProps = Object.entries(props).reduce((ps, [k, v]) => {
      if (prev.current[k] !== v) ps[k] = [prev.current[k], v];
      return ps;
    }, {});
    if (Object.keys(changedProps).length > 0) {
      console.log(`[Re-render] ${componentName}:`, changedProps);
    }
    prev.current = props;
  });
}


// Usage in your component:
// useTraceUpdate('DraggableCard', { id, queryData, urlParam });
```


#### B. Prefix Logging for Sub-systems
Create a standard logger to easily filter your console:
```javascript
export const logger = {
  dnd: (...args) => console.log('🟢 [DND-KIT]', ...args),
  query: (...args) => console.log('🟠 [TANSTACK]', ...args),
  nuqs: (...args) => console.log('🔵 [NUQS]', ...args),
};
```


#### C. dnd-kit Global Monitor
Instead of placing `console.log` in every draggable item, hook into dnd-kit's global lifecycle using `useDndMonitor`:
```javascript
import { useDndMonitor } from '@dnd-kit/core';


function DndLogger() {
  useDndMonitor({
    onDragStart(event) { logger.dnd('Drag Start', event.active.id); },
    onDragMove(event) { /* Optional: can be noisy */ },
    onDragOver(event) { logger.dnd('Drag Over', event.over?.id); },
    onDragEnd(event) { logger.dnd('Drag End', event.active.id, '->', event.over?.id); },
    onDragCancel(event) { logger.dnd('Drag Canceled', event.active.id); },
  });
  return null;
}
// Render <DndLogger /> inside your <DndContext>
```

### Step 3: Isolation Strategy (The Debugging Process)


To find the root cause, you need to isolate the libraries. Follow this checklist:


1.  **Is it a Nuqs (URL) issue?**
    *   *Test:* Hardcode the query parameters instead of using `useQueryState`. Does the bug disappear?
    *   *Symptom:* If updating the URL causes a hard page refresh or a total loss of React state, your router might not be configured correctly for shallow routing, or `nuqs` is causing the `DndContext` to unmount.
2.  **Is it a TanStack Query issue?**
    *   *Test:* Mock the data. Replace `useQuery` with a static `const data = [...]`. Does the bug disappear?
    *   *Symptom:* The UI works, but jumps suddenly. This usually means an Optimistic Update failed, or a background `refetchOnWindowFocus` fired during a drag operation, replacing the data array and destroying dnd-kit's DOM nodes.
3.  **Is it a dnd-kit issue?**
    *   *Test:* Remove Nuqs and TanStack query integrations temporarily from the drag event handlers.
    *   *Symptom:* Items won't drag, or drop areas aren't detected. This almost always means `setNodeRef` is not being applied to the correct underlying DOM element (usually broken by custom components).


---


### Step 4: Check for Stack-Specific "Gotchas"


Because of how these three libraries interact, there are highly specific bugs that occur. Check if your problem matches one of these:


#### Gotcha 1: Nuqs URL updates breaking dnd-kit (The "Dropped Drag" Bug)
*   **The Problem:** You update a `nuqs` URL parameter during `onDragStart` or `onDragMove`. Nuqs updates the URL, which triggers your router/layout to re-render. This re-render unmounts the `<DndContext>` or the draggable item, causing the drag to suddenly cancel.
*   **The Fix:** **Defer URL updates.** Never update a URL state via nuqs *during* an active drag. Save the state in a local `useState` during the drag, and only push it to `nuqs` (the URL) inside the `onDragEnd` handler.
*   **Alternative Fix:** Ensure you are using `shallow: true` (if supported by your router framework) in nuqs so it doesn't trigger layout re-renders.


#### Gotcha 2: TanStack Query background refetches destroying DOM Refs
*   **The Problem:** While dragging an item, TanStack Query decides to refetch the list (e.g., you clicked out of the window and back, triggering `refetchOnWindowFocus`). The query finishes, replaces the list array with a "new" array, React re-renders the list, and dnd-kit's DOM nodes are destroyed and recreated. The drag stops mid-air.
*   **The Fix:** Pause background refetches during drags, or rely on stable keys.
    ```javascript
    // Use stable IDs in your map:
    {data.map(item => <Draggable key={item.id} id={item.id} />)} 
    // DO NOT use index: <Draggable key={index} ... />
    ```


#### Gotcha 3: Stale Closures in `onDragEnd`
*   **The Problem:** Your `onDragEnd` function calls a TanStack `mutate()` or updates a `nuqs` URL, but it uses an old version of the state because the function was memoized improperly.
*   **The Fix:** Always use functional updates, or ensure your `useMutation` callbacks rely on the variables passed directly into the event, rather than state from the component scope.
    ```javascript
    const handleDragEnd = (event) => {
       const { active, over } = event;
       // Rely on event.active.id, NOT a component-level state variable
    }
    ```


#### Gotcha 4: Ref Forwarding Mismatches
*   **The Problem:** `dnd-kit` requires direct access to DOM nodes to measure them and apply transforms. If you wrap a draggable item in a complex custom component, the ref might get lost.
*   **The Fix:** Ensure you are using `forwardRef` properly in all custom components involved in dragging.
    ```javascript
    const MyCustomCard = forwardRef(({ children, ...props }, ref) => (
      <div ref={ref} {...props}>
         {children}
      </div>
    ));
    ```


### Summary of your Next Steps:
1. Add the `<DndLogger />` and the `useTraceUpdate` hook to your Draggable items.
2. Open your console and start a drag. 
3. Look at the exact sequence: *Did a Query refetch happen? Did Nuqs update the URL? Did `useTraceUpdate` fire immediately before the drag failed?* 
4. Once you see the order of events in the console, you will know exactly which library is interrupting the other.

## 📋 Step 3: TanStack Query — Specific Debug Steps


### 3a. Inspect Query Keys

```ts
useQuery({
```

```ts
const queryClient = new QueryClient({
```

## 🖱️ Step 4: DND Kit — Specific Debug Steps


The `DragDropProvider` component wraps your draggable and droppable elements, handles the drag and drop interactions, and allows you to listen to drag and drop events.


### 4a. Log Every DND Event
Instrument all event handlers on your `DndContext`/`DragDropProvider`:
```tsx
<DndContext
  onDragStart={(e) => console.log('[DND] dragStart', e)}
  onDragOver={(e) => console.log('[DND] dragOver', e)}
  onDragEnd={(e) => console.log('[DND] dragEnd', e)}
  onDragCancel={(e) => console.log('[DND] dragCancel', e)}
>
```


### 4b. Verify Item IDs Are Unique
A critical DND Kit requirement: every draggable and droppable must have a **unique `id`**. A common bug is:
```tsx
// ❌ Bug: duplicate IDs cause items to move in unison
items.map(item => <Draggable id="item" />)


// ✅ Correct
items.map(item => <Draggable id={item.id} />)
```


### 4c. Debug the `onDragEnd` Handler
Use console logs or debugging tools to track the values of `result.source` and `result.destination` in the `onDragEnd` function.


```ts
onDragEnd={(event) => {
  const { active, over } = event
  console.log('[DND] active:', active?.id, 'over:', over?.id)
  if (!over) {
    console.warn('[DND] No valid drop target — skipping state update')
    return
  }
  // ... your logic
}}
```


### 4d. Check Edge Cases
Edge cases often reveal hidden issues. Test scenarios like dragging an item outside the drop zone or dropping it in an invalid area.


### 4e. Sensor Debugging
If drag doesn't activate properly, check sensor config. Add activation constraints to prevent accidental drags:
```ts
import { useSensor, useSensors, PointerSensor } from '@dnd-kit/core'


const sensors = useSensors(
  useSensor(PointerSensor, {
    activationConstraint: { distance: 8 }, // must move 8px to activate drag
  })
)
```

## 🔗 Step 5: nuqs — Specific Debug Steps


nuqs provides a familiar `React.useState`-like API that syncs with the URL, and a `useQueryStates` hook to manage multiple keys at once.


### 5a. Enable Built-in Logging (as mentioned in Step 1)
```js
localStorage.debug = 'nuqs'
```
This will print every URL state read/write to the console.


### 5b. Inspect the URL Directly
Since nuqs ensures that the URL reflects the current application state, the browser address bar *is* your state inspector. Manually edit the URL to test different states and verify your parsers handle them correctly.


### 5c. Verify Your Parsers
nuqs includes built-in parsers that transform URL strings into typed values such as integers, booleans, and dates, ensuring type safety throughout the application.


Log parsed values immediately after reading them:
```ts
const [page, setPage] = useQueryState('page', parseAsInteger.withDefault(1))
console.log('[NUQS] page parsed as:', page, typeof page)
```


### 5d. Test with the nuqs Testing Adapter
Use the provided test adapter to test your components in isolation.


```tsx
import { NuqsTestingAdapter } from 'nuqs/adapters/testing'


render(<YourComponent />, {
  wrapper: ({ children }) => (
    <NuqsTestingAdapter searchParams="?page=2&filter=active">
      {children}
    </NuqsTestingAdapter>
  )
})
```


### 5e. Watch for Browser Rate Limiting
Nuqs's "time safety" feature decouples UI updates from URL updates, preventing browser rate-limiting issues. Throttling and debouncing capabilities also prevent excessive network requests during high-frequency interactions like text inputs.


If your URL updates seem delayed or dropped, use:
```ts
const [value, setValue] = useQueryState('q', {
  ...parseAsString,
  limitUrlUpdates: debounce(300),
})
```

## 🔄 Step 6: Debug Cross-Library Interactions


This is where the hardest bugs live. The three libraries interact in specific, common patterns:


### Pattern: nuqs drives TanStack Query keys
```ts
const [filter] = useQueryState('filter', parseAsString.withDefault('all'))


// If filter changes, this re-fetches automatically
const { data } = useQuery({
  queryKey: ['items', filter],
  queryFn: () => fetchItems(filter),
})


// Debug: log both together
console.log('[SYNC] filter:', filter, '| query status:', status)
```


### Pattern: DND updates that then invalidate TanStack Query
```ts
onDragEnd={async (event) => {
  const { active, over } = event
  if (!over) return


  console.log('[DND→QUERY] Reordering', active.id, 'to', over.id)


  await reorderMutation.mutateAsync({ from: active.id, to: over.id })


  console.log('[DND→QUERY] Invalidating list cache')
  queryClient.invalidateQueries({ queryKey: ['items'] })
}}
```


### Pattern: nuqs param + DND state conflict
A common bug: URL state and local drag state get out of sync. Always use **one source of truth**:
```ts
// ✅ Let nuqs own the order, DND only triggers updates
const [itemOrder, setItemOrder] = useQueryState('order', parseAsArrayOf(parseAsString))


onDragEnd={(event) => {
  const newOrder = computeNewOrder(itemOrder, event)
  console.log('[DND→NUQS] New order:', newOrder)
  setItemOrder(newOrder)
}}
```

## ✅ Step 8: Systematic Elimination Checklist


Work through this checklist in order:


1. **Reproduce it minimally** — Comment out DND and nuqs, does the data bug still exist? ➡️ It's a TanStack Query issue.
2. **Comment out nuqs** — Hard-code URL params. Does the bug go away? ➡️ It's a nuqs/URL state issue.
3. **Comment out TanStack Query** — Use static data. Does the drag still break? ➡️ It's a pure DND Kit issue.
4. **Check for stale closures** — Are your DND event handlers reading stale state? Use `useCallback` with correct deps.
5. **Check re-render timing** — Is a re-render from TanStack Query resetting your DND drag mid-action? Wrap your component in `React.memo` and check.
6. **Check for React StrictMode double-invocation** — In dev mode, effects run twice. This can cause duplicate mutations or double state updates.

## 🧪 Bonus: Reusable Debug Utility Hook


```ts
function useDebugState<T>(label: string, value: T) {
  const prevRef = useRef<T>(value)


  useEffect(() => {
    if (prevRef.current !== value) {
      console.log(`[DEBUG:${label}] changed:`, {
        from: prevRef.current,
        to: value,
      })
      prevRef.current = value
    }
  })
}


// Usage
useDebugState('filter', filter)
useDebugState('queryStatus', status)
useDebugState('dndActive', activeId)
```


This hook tells you exactly **when and what changed** across all three systems.
